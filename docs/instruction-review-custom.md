# 単一ファイル向けカスタマイズレビュー観点（サンプル）

このプロンプトでは 1 回の実行につき 1 ファイルのみをレビュー対象とします。
Web アプリの複雑な Controller や大規模テストクラスは扱わず、演習問題のようなシンプルなコードを対象に以下の観点を適用してください。

## TypeScript / JavaScript 演習

### `src/exercises/Step01_HelloComponent.tsx`
- JSX 内で渡された `props` の型（`HelloProps` など）が厳密に定義されていること。
- 表示テキストに直接結合したユーザー入力がある場合、エスケープまたは検証が行われていること。
- スタイルは `styles.ts` など共通ファイルを参照し、コンポーネント内でのハードコードを避けていること。

### `src/exercises/Step02_NumberUtils.ts`
- 数値計算の境界値（0、負数、最大値）がユニットテストコメントで示されているか、少なくとも関数内でチェックされていること。
- Promise/async 関数は `try/catch` を持ち、`reject` ではなく明示的なエラーを投げていること。
- `console.log` 等のデバッグ用途コードが残っていないこと。

## Java 演習

### `Task01PrintDate.java`
- `LocalDate`/`DateTimeFormatter` を使い、現在日付と指定日付との差分が正しく計算されていること。
- 日本語表示が必要な場合は `Locale.JAPAN` を明示的に指定していること。
- 標準出力にフォーマット済み文字列のみを出し、デバッグ用出力を残していないこと。

### `Task02ArrayStats.java`
- `int[]` などの入力配列が空の場合のハンドリング（早期 return, 例外など）があること。
- 平均値や最大値の計算でオーバーフローを避けるため `long` あるいは `double` を用いていること。
- `for` ループ・拡張 `for` ループのどちらか一方に統一し、不要な重複処理がないこと。

### `Task03SimpleGame.java`
- `Scanner` や `BufferedReader` のクローズ処理が `try-with-resources` で書かれていること。
- 乱数の範囲（例: 1〜6）は要件通りかつ複数回のプレイで再利用される `Random` インスタンスを保持していること。
- 入力値チェック（数字以外が入力された場合など）が実装されていること。

---

> 上記はサンプルです。実際の演習ファイルに合わせてファイル名や観点をカスタマイズしてください。
